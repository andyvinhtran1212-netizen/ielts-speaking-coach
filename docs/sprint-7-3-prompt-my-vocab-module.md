## Phase A — Module API + refactor plan (1h)

### Step 1: Module API contract

Code drafts the canonical module shape that **every** vocab module (and future similar) will implement:

```js
// /js/vocab-modules/my-vocab.js  (and flashcards.js, exercises.js in later sprints)

/**
 * @param {HTMLElement} container — element where the module renders its body.
 * @param {Object} opts
 * @param {boolean} opts.embedded — true when mounted by vocabulary.html parent; false when standalone shell.
 * @returns {Promise<{ unmount: () => void }>}
 */
export async function mount(container, opts = {}) {
  // 1. Inject template literal HTML into container.innerHTML
  // 2. Wire event listeners (delegation preferred over global window.* handlers).
  // 3. Bootstrap state (auth, fetch initial data).
  // 4. Return { unmount } to clean up listeners + clear container.
}
```

Decisions to make:
- **Inline `onclick=""` strategy**: convert to `data-action="markFixed"` + delegated click listener? OR keep window globals but scope under `window.__vocabModules.myVocab.*` to avoid collision when 3 modules co-mount?
- **Auth redirect behavior**: when `opts.embedded === true` AND no auth, should the module redirect the parent (`window.top.location.href = '/index.html'`), or render a "please log in" inline state? Recommendation: **redirect** (auth loss = top-level event; rendering inline state in iframe-less embedded context is awkward).
- **Re-mount behavior**: if `mount()` is called twice on the same container, should the second call replace? Recommendation: **idempotent** — second `mount()` is a no-op if container already mounted; module exposes `data-mounted` attribute as the guard.

### Step 2: Parent loader infrastructure

`frontend/js/vocab-landing.js` currently sets `iframe.src` on tab activation. Refactor to:

```js
// vocab-landing.js (post-Sprint 7.3)
const TAB_LOADERS = {
  'my-vocab': () => import('/js/vocab-modules/my-vocab.js'),
  // flashcards + exercises stay on iframe path this sprint (loaded via TAB_SOURCES iframe src)
};

async function activateTab(tabName) {
  // ... existing tab UI toggle ...

  if (TAB_LOADERS[tabName]) {
    const containerEl = document.querySelector(`#tab-${tabName}-mount`);
    if (!containerEl.hasAttribute('data-mounted')) {
      showSkeleton(containerEl);
      try {
        const mod = await TAB_LOADERS[tabName]();
        await mod.mount(containerEl, { embedded: true });
        containerEl.setAttribute('data-mounted', 'true');
      } catch (err) {
        showImportError(containerEl, err);
      }
    }
  }
  // ... fall through to existing iframe path for flashcards + exercises ...
}
```

## Strategic notes

- **Module loader is the foundation contract** every later phase (7.4, 7.5) will follow. Get the API right this sprint — changing it later costs 2× rework.
- **Phased rollout means parent must support BOTH paths simultaneously.** `vocab-landing.js` `activateTab()` branches: module path for my-vocab, iframe path for flashcards + exercises during Sprint 7.3 + 7.4.
- **Don't retire `embedded-mode.css` yet** — flashcards + exercises still depend on it through Sprint 7.5.
- Test pin churn is bounded to chrome-unification + new module-loader file. Other test suites untouched.
- 11 `window.*` handlers from inline-onclick: prefer **event delegation** for cleaner module boundary, but scoped-globals is acceptable if delegation breaks card-action handlers in a way that's expensive to rewrite. Phase B chooses.
